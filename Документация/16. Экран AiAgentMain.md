# Экран AiAgentMain
Этот файл описывает экран `AiAgentMain`, который расширяет `AiStateAgent` интеграцией MCP-инструментов в execution-фазе.
Актуализировано: 01.04.2026.

## Назначение экрана
`AiAgentMain` — центральный экран «инструментального агента», где модель:
- выполняет state-machine цикл с валидацией;
- вызывает внешние MCP-инструменты по `tool_request`;
- обрабатывает агентные slash-команды (`/help`, `/review-pr`, `/auth-token`);
- использует RAG-контекст как в обычном диалоге, так и в сценариях PR-review.

## Ключевые функции
- Полный state-machine цикл (`Planning`, `Execution`, `Checking`, `Done`).
- Поддержка всех функций `AiStateAgent`:
  - планирование;
  - валидации;
  - инварианты;
  - recovery-ветки;
  - long-term память.
- MCP-подсистема:
  - выбор MCP-серверов;
  - кэширование инструментов;
  - выполнение tool-вызовов.
- Парсинг `tool_request` в execution-ответе.
- Поддержка WebSocket-сценариев и логирования MCP-диалога.
- Хранение app-level настроек в `app_settings`.
- Встроенный agent mode и команды:
  - `/help [вопрос]` — агентный ответ с комбинированием MCP/RAG контекста;
  - `/review-pr <repo_url> [pr_number] [--publish]` — структурированное AI-ревью pull request;
  - `/auth-token <repo_url|owner:ORG|global> <token> [token_type]` — сохранение GitHub токена через MCP.
- PR-review интеграция:
  - загрузка PR-контекста через MCP-инструмент `git.pr_review_context`;
  - публикация результата в PR через `scm.submit_pr_review` (опционально);
  - автоматическая попытка `auth.upsert_token`, если сервер возвращает диагностику по отсутствующему токену.
- Гибридный контекст для ревью:
  - semantic RAG по локальной embedding-базе;
  - lexical fallback по `Документация/*.md` и содержимому измененных PR-файлов.

## Зависимости
- База: `createAiAgentMainDatabase(AiAgentMainDatabaseDriverFactory())`.
- SQLDelight-схема `aiagentmaindb` + `app_settings`.
- MCP сервис: `RemoteMcpService`.
- Prompt-шаблоны:
  - `AiAgentMainPlanningPrompt`;
  - `AiAgentMainExecutionPrompt`;
  - `AiAgentMainValidationPrompt`.
- Сетевые LLM-клиенты.
- Agent-компоненты:
  - `AgentPrReviewUseCase`, `AgentMcpPrReviewGitGateway`, `AgentAuthTokenUseCase`;
  - `AgentSlashCommandParser` и модели команд;
  - `AiAgentMainPrReviewRag` и `AiAgentMainRagRetrieval`.

## Используемые данные
- Базовые чатовые и профильные таблицы.
- Инварианты и память.
- Настройки MCP и служебные флаги приложения.
- Журнал MCP-событий, привязанный к чатам/веткам.

## Логика MCP-вызова
- На execution-шаге модель может вернуть структуру `tool_request`.
- Экран валидирует её и определяет целевой сервер/endpoint.
- Вызывается `RemoteMcpService`:
  - `callTool`;
  - `callToolViaWebSocket`;
  - `callToolWithAuxWebSocket` (по необходимости).
- Результат инструмента добавляется в контекст и execution-шаг повторяется с учетом новых данных.

## Логика PR-review команды
1. Команда `/review-pr` парсится в `AgentPrReviewTarget`.
2. Через MCP-шлюз запрашивается PR-контекст (diff, список файлов, метаданные PR).
3. Собирается смешанный RAG-контекст:
  - semantic retrieval по embedding-чанкам;
  - lexical ранжирование по документации проекта и файлам PR.
4. Модель вызывается со строгим JSON-контрактом (`potential_bugs`, `architecture_issues`, `recommendations`, `summary`, `confidence`).
5. При `--publish` выполняется публикация ревью-комментария в PR через MCP.

## Логика команды `auth-token`
- Команда `/auth-token` нормализует цель привязки токена (`repo`/`owner`/`global`).
- Выполняется вызов `auth.upsert_token` на активном MCP-сервере.
- Возвращаются диагностические сообщения сервера и статус сохранения токена.

## Возможности
- Реальный цикл «план + инструмент + проверка результата».
- Поддержка асинхронных WebSocket-инструментов.
- Локальный контроль правил (инварианты) поверх внешних инструментальных вызовов.

## Недостатки и ограничения
- Сценарий чувствителен к формату `tool_request` и точности endpoint.
- Увеличенная сложность состояний по сравнению с `AiStateAgent`.
- Большой объем ответственности внутри одного экрана, несмотря на выделение helper/use-case файлов.
- Повышенная трудоемкость ручной отладки при одновременных ошибках LLM и MCP.
- Качество PR-review зависит от полноты MCP Git-инструментов и доступности токена/прав.
- Гибридный RAG требует актуального embedding-корпуса: при устаревших данных semantic-часть теряет полезность.
