# Потоки сценариев
Этот файл описывает сквозные пользовательские и системные сценарии, включая обработку данных на каждом этапе.
Актуализировано: 01.04.2026.

## Сценарий 1: обычный чат (App/AiAgent/AiWeek3)
1. Пользователь вводит запрос и выбирает API/модель.
2. Экран собирает `DeepSeekChatRequest` с учетом системного промпта и параметров генерации.
3. Запрос отправляется в выбранный API-клиент.
4. Поток ответа обновляет временное сообщение в UI.
5. По завершении фиксируется финальный ответ и метрики (`response_time`, токены при наличии).
6. В stateful-экранах запись сохраняется в БД, обновляется активная ветка и связанные флаги.

## Сценарий 2: контекстные стратегии (`AiAgent`/`AiWeek3`)
Перед отправкой нового запроса экран выбирает контекст:
- `Summarization`: периодически генерируется краткая выжимка прошлых сообщений.
- `SlidingWindow`: берется только последние N сообщений.
- `StickyFacts`: отдельным запросом выделяются устойчивые факты, затем подмешиваются в system-контекст.
- `Branching`: запрос и ответы привязываются к выбранной ветке истории.

## Сценарий 3: profile + long-term memory (`AiWeek3`)
1. Пользователь выбирает или создаёт профиль.
2. Профиль определяет активные фичи стратегии контекста.
3. Память профиля (`memory_entries`) редактируется вручную.
4. При генерации ответа long-term память добавляется в системную часть контекста (если фича включена).

## Сценарий 4: подготовка базы эмбеддингов (`EmbedingGeneration`)
1. Пользователь выбирает файлы и стратегии чанкинга.
2. Система извлекает текст из файла (для PDF через PDFBox).
3. Текст делится на чанки выбранными стратегиями.
4. Для каждого чанка генерируется embedding (локальный endpoint).
5. Чанк + embedding + метаданные секции сохраняются в `embedding_chunks`.

## Сценарий 5: RAG-ответ (`AiAgentRAG`)
1. Пользовательский запрос обновляет `RagTaskState` (цель, уточнения, ограничения, термины).
2. При `useRewrite` выполняется эвристическое переписывание retrieval-запроса.
3. Генерируется embedding запроса.
4. Все сохраненные embedding векторы ранжируются по cosine similarity.
5. Применяются `topKBefore`, затем фильтр порога, затем `topKAfter`.
6. Формируется retrieval-контекст и диалоговая подложка.
7. Модель генерирует ответ.
8. Ответ нормализуется в строгий формат со ссылками на источники и цитатами.

## Сценарий 6: state-agent (`AiStateAgent`)
1. Планирование: модель получает промпт на построение JSON-плана.
2. Выполнение: по шагам выполняется execution-промпт.
3. Проверка:
  - формальная (структура/формат/критерии),
  - семантическая (полнота/согласованность/точность).
4. При проблеме формируется `ValidationFailureInfo`, создается recovery-ветка.
5. Выполнение перезапускается с нужного шага.
6. При успешной проверке результат переносится в основную ветку.

## Сценарий 7: state-agent + MCP (`AiAgentMain`)
1. План формируется аналогично state-agent сценарию.
2. Во время execution парсится `tool_request`.
3. Если модель запросила tool-вызов:
  - выбирается MCP-сервер и endpoint;
  - вызывается инструмент через `RemoteMcpService`;
  - результат инструмента добавляется в execution-контекст и шаг выполняется повторно.
4. Дальше запускается валидационный цикл, включая проверку инвариантов.

## Сценарий 8: агентные команды (`AiAgentMain`, agent mode)
1. Пользователь вводит slash-команду (`/help`, `/review-pr`, `/auth-token`).
2. Команда парсится через `AgentSlashCommandParser`, проверяются аргументы.
3. Для `/help`:
  - агент строит план источников, комбинирует MCP-данные и RAG-контекст;
  - формирует ответ с рекомендациями или диагностикой.
4. Для `/review-pr`:
  - через MCP запрашивается контекст PR (`git.pr_review_context`);
  - строится гибридный контекст: semantic RAG + lexical PR-RAG;
  - модель возвращает строго структурированное JSON-ревью;
  - при флаге `--publish` выполняется публикация через `scm.submit_pr_review`.
5. Для `/auth-token`:
  - вызывается MCP-инструмент `auth.upsert_token`;
  - токен связывается с `repo`, `owner` или `global` в зависимости от аргументов.
