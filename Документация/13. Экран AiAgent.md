# Экран AiAgent
Этот файл описывает экран `AiAgent`: его назначение, сценарии, структуру данных и ограничения.

## Назначение экрана
`AiAgent` — основной stateful-чат проекта, в котором поверх базового обмена сообщениями реализованы стратегии управления контекстом и ветвление истории.

## Ключевые функции
- Хранение чатов и сообщений в отдельной БД (`app.db`).
- Управление несколькими чатами.
- Управление ветками диалога (branching) и их хранение в БД.
- Переключение между `raw` и `real` потоками сообщений.
- Поддержка контекстных стратегий:
  - `Summarization`;
  - `SlidingWindow`;
  - `StickyFacts`;
  - `Branching`.
- Поддержка всех LLM-провайдеров и локальной LLM.
- Отображение токенов и времени ответа в `paramsInfo`.

## Зависимости
- База: `createAppDatabase(DatabaseDriverFactory())`.
- SQLDelight-запросы таблиц: `chats`, `chat_messages`, `chat_feature_state`, `chat_branch_messages`.
- Сетевые клиенты LLM (те же, что на экране `App`).
- Общие модели запроса/ответа сети.

## Какие данные использует
- Персистентные:
  - список чатов;
  - история сообщений чата;
  - состояние feature-флагов по чату;
  - сообщения веток.
- Временные:
  - текущее сообщение пользователя;
  - выбранный API и модель;
  - текущая активная ветка;
  - промежуточные стрим-чанки ответа.

## Логика экрана
- При открытии чата загружается история, настройки фич и ветки.
- При отправке сообщения формируется контекст согласно выбранной стратегии.
- На этапе стрима одновременно ведутся:
  - обновление UI;
  - запись финализированного ответа в `chat_messages`;
  - запись в ветку (если ветвление активно).
- Для `StickyFacts` выполняется дополнительный LLM-запрос на извлечение устойчивых фактов.
- Для `Summarization` по порогу токенов выполняется отдельный запрос суммаризации.

## Возможности
- Гибкая настройка контекста на уровне отдельного чата.
- Разделение наблюдаемой и «рабочей» ленты (`show_raw_history`).
- Удобное исследование влияния стратегии контекста на ответы модели.

## Недостатки и ограничения
- Экран очень крупный, логика и UI тесно связаны.
- Много локального состояния, из-за чего высокая сложность поддержки.
- Стратегии контекста реализованы прямо в экране, без выделенного domain-слоя.
- При изменении схемы БД возможен сброс данных (из-за стратегии пересоздания базы).
