# Рекомендации по развитию
Этот файл содержит практические предложения по улучшению проекта с учетом текущей структуры.

## 1. Разделить orchestration и UI
- Вынести логику сценариев из экранов в отдельные use-case/сервисные классы.
- Оставить в composable только представление и обработку событий.
- Это снизит связанность и ускорит доработки.

## 2. Ввести тестовый каркас
- Добавить unit-тесты для:
  - построения контекста (summary/sliding/sticky/memory);
  - RAG-ранжирования и форматирования источников;
  - парсинга `tool_request` и валидационных ответов;
  - проверки инвариантов.
- Добавить интеграционные тесты SQLDelight-запросов.

## 3. Перейти от «пересоздания БД» к миграциям
- Даже минимальные миграции для сохранения ключевых таблиц снизят риск потери данных.
- По структуре проекта видно, что это приоритетная зона.

## 4. Унифицировать и переиспользовать chunking-логику
- Оставить единую реализацию в одном пакете.
- Экрану `EmbedingGeneration` дать тонкий orchestration-слой поверх общих стратегий.

## 5. Сделать явный модуль конфигурации провайдеров
- Централизовать:
  - список моделей;
  - требования к API-ключам;
  - маппинг параметров.
- Это уменьшит дублирование между экранами.

## 6. Укрепить наблюдаемость
- Добавить структурированное логирование шагов агента и MCP-вызовов.
- Добавить единый формат ошибок для UI и журнал событий выполнения.

## 7. Подготовить переносимость KMP-логики
- Последовательно выносить domain/data-часть из `jvmMain` в `commonMain`.
- В первую очередь перенести reusable бизнес-процессы: state-machine, валидация, RAG orchestration.

## 8. Очистить вопросы кодировки строк
- Провести нормализацию кодировок исходников.
- Проверить отображение русских строк в UI и логах на всех целевых ОС.
