# Экран AiStateAgent
Этот файл описывает экран `AiStateAgent`, где реализован управляемый цикл «планирование → выполнение → проверка».

## Назначение экрана
`AiStateAgent` — экспериментальный state-machine агент, который не просто отвечает на сообщение, а проходит фиксированный жизненный цикл решения задачи с контролем качества и восстановлением после ошибок.

## Ключевые функции
- Стадии `AgentState`:
  - `Idle`;
  - `Planning`;
  - `Execution`;
  - `Checking`;
  - `Done`.
- Генерация структурированного JSON-плана.
- Пошаговое выполнение плана.
- Двухфазная проверка:
  - формальная;
  - семантическая.
- Редактирование плана пользователем.
- Проверка инвариантов.
- Ветвь восстановления при неуспешной проверке.
- Панели инвариантов и long-term памяти.

## Зависимости
- База: `createAiStateAgentDatabase(AiStateAgentDatabaseDriverFactory())`.
- SQLDelight-схема `aistateagentdb`:
  - чатовые таблицы;
  - `memory_entries`;
  - `invariant_entries`.
- Prompt-шаблоны:
  - `AiStatePlanningPrompt`;
  - `AiStateExecutionPrompt`;
  - `AiStateValidationPrompt`.
- Сетевые LLM-клиенты.

## Данные экрана
- Персистентные:
  - история чатов/веток;
  - память;
  - инварианты.
- Операционные:
  - `planText`;
  - `executionText`;
  - результат проверок;
  - текущая ветка и текущая стадия.

## Логика жизненного цикла
- `Planning`: модель возвращает план в JSON; если данных недостаточно — запрашиваются уточнения.
- `Execution`: шаги плана исполняются последовательно, с накоплением промежуточных результатов.
- `Checking`: запускается формальная и семантическая валидация.
- При ошибке формируется `ValidationFailureInfo`, создается recovery-ветка и задается точка повторного запуска.
- При успехе ветка результата переносится в основную.

## Возможности
- Прозрачный pipeline принятия решения.
- Контролируемые точки перезапуска.
- Отдельные ветки этапов для аудита процесса.
- Явные инварианты как проверяемые ограничения.

## Недостатки и ограничения
- Высокая сложность orchestration-кода.
- Надежность зависит от строгого соблюдения JSON-контрактов моделью.
- Инварианты частично проверяются эвристически по тексту.
- При крупном изменении промптов возможна деградация стабильности этапов.
