# Архитектура и платформенное разделение
Этот файл объясняет архитектурный подход проекта, распределение ответственности между слоями и `source set`.

Архитектура проекта практическая и экранно-ориентированная: значительная часть бизнес-логики находится непосредственно в экранных composable-функциях и связанных локальных helper-функциях. Это не «чистая» слоистая архитектура в строгом виде, но в коде все же можно выделить устойчивые уровни.

## Архитектурные уровни
- UI и orchestration: крупные экраны `AiAgentScreen`, `AiWeek3Screen`, `AiStateAgentScreen`, `AiAgentMainScreen`, `AiAgentRAGScreen`, `EmbedingGenerationScreen`, `AppScreen`, `AiAgentMCPScreen`.
- Сетевой слой: `NetworkClient`, `DeepSeekApi`, `OpenAiApi`, `GigaChatApi`, `ProxyOpenAiApi`, `LocalLlmApi`, `StreamingChatParser`.
- Интеграция инструментов: `RemoteMcpService`.
- Данные и хранилище: SQLDelight-схемы и фабрики БД (`DatabaseFactory`, `AiWeek3...`, `AiStateAgent...`, `AiAgentMain...`, `AiAgentRag...`, `EmbedingGeneration...`).
- Конфигурационный слой: build-скрипты, генерация `BuildSecrets`, `secrets.properties`, `Preferences`.

## Kotlin Multiplatform в проекте
### `commonMain`
- Общие API-клиенты и модели запросов/ответов.
- Общие `expect`-абстракции для HTTP engine и драйверов БД.
- SQLDelight `.sq` схемы и запросы.
- MCP сервис.

### `jvmMain`
- Все экраны и весь пользовательский интерфейс.
- `actual`-реализации для HTTP engine (`CIO`) и драйверов SQLite.
- Локальные утилиты обработки текста/PDF/чанкинга/эмбеддингов.
- Точка входа приложения (`main.kt`) и маршрутизация между экранами (`RootApp.kt`).

### Что это значит для сопровождения
- Общий код уже подготовлен для переноса сетевых и data-компонентов на другие платформы.
- Фактическая бизнес-логика сейчас существенно завязана на `jvmMain`, поэтому перенос на другие targets потребует выделения domain/use-case уровня из экранов.

## Межслойные взаимодействия
- Экран формирует параметры запроса и историю сообщений.
- Экран вызывает API-клиент выбранного провайдера.
- Потоковый ответ приходит через `onChunk`, UI обновляет временное сообщение.
- Финальный ответ и метаданные сохраняются в SQLDelight.
- Дополнительные стратегические шаги (summary, sticky facts, validation, RAG retrieval, MCP-вызовы) выполняются в той же orchestrating функции экрана.

## Навигация
- Навигация централизована через `RootScreen` и `RootApp`.
- Выбор экрана сохраняется в `Preferences` (`last_opened_root_screen`), чтобы восстановить последнее состояние при следующем запуске.
