# Экран App
Этот файл описывает экран `App`: его функциональность, внутреннюю логику, зависимости, используемые данные и ограничения.

## Назначение экрана
`App` — базовый и самый простой чатовый экран в проекте. Он нужен как «операционный стенд» для ручной проверки разных LLM API без сложной агентной логики, профилей, памяти и ветвления.

## Ключевые функции
- Выбор провайдера (`DeepSeek`, `OpenAI`, `GigaChat`, `ProxyAPI`, `Local LLM`).
- Выбор модели из списка для выбранного провайдера.
- Ввод системного промпта.
- Ввод параметров генерации:
  - `temperature`;
  - `top_p`;
  - `max_tokens`;
  - `presence_penalty`;
  - `frequency_penalty`;
  - `stop`.
- Потоковый вывод ответа в реальном времени.
- Отображение параметров запроса и времени ответа.

## Зависимости
- Сетевые клиенты: `DeepSeekApi`, `OpenAiApi`, `GigaChatApi`, `ProxyOpenAiApi`, `LocalLlmApi`.
- Общие сетевые модели: `DeepSeekChatRequest`, `DeepSeekMessage`.
- Конфигурация секретов: `BuildSecrets` + fallback на переменные окружения.
- UI-стек: Compose Material 3.

## Какие данные использует экран
- Внутреннее состояние в памяти процесса:
  - список сообщений;
  - текущие параметры запроса;
  - выбранный API и модель;
  - флаг загрузки.
- Экран не использует SQLDelight и не сохраняет историю в БД.

## Логика работы
- При отправке пользовательского сообщения формируется единый `DeepSeekChatRequest`.
- В зависимости от выбранного API вызывается соответствующий клиент.
- Ответ принимается потоково через callback чанков.
- В процессе стрима постоянно обновляется «ассистентское» сообщение.
- После завершения стрима записывается финальный текст и итоговое `response_time`.

## Сильные стороны
- Минимальный порог входа: быстро проверить доступность API и модели.
- Унифицированный интерфейс работы с несколькими провайдерами.
- Удобно для ручной диагностики качества и времени ответа.

## Недостатки и ограничения
- Нет постоянного хранения истории диалогов.
- Нет расширенных возможностей контекста (память, профили, суммаризация, ветвление).
- Нет встроенного механизма перезапуска сценария с промежуточного шага.
- Логика ориентирована на ручное использование, не на сложные агентные пайплайны.
